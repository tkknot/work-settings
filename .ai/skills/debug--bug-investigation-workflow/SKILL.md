# バグ調査ワークフロー

Backlog連携を起点とした体系的なバグ調査・修正の進め方ガイドです。

## 人間 / AI 役割分担

| 役割 | 担当 |
|------|------|
| **ビジネスインパクトの判断**: バグの緊急度・ユーザーへの影響範囲を評価する | 人間 |
| **修正方針の決定**: Quick Fix（応急処置）vs Deep Fix（根本解決）のトレードオフ判断 | 人間 |
| **Backlogの確認**: MCP経由で課題内容を取得し、現象・環境・期待値を要約する | AI |
| **原因・影響範囲の特定**: コードリーディングでバグの発生原因と影響範囲を特定する | AI |
| **再現コード生成**: バグを再現するテストケースを自動生成する | AI |
| **修正案の提示**: Quick Fix / Deep Fix の複数パターンを比較提示する | AI |

---

## Quick Fix vs Deep Fix 判断基準

| 観点 | Quick Fix（応急処置） | Deep Fix（根本解決） |
|------|---------------------|-------------------|
| 修正範囲 | 問題箇所のみ最小限修正 | 設計レベルで対処 |
| リスク | 低い（影響が限定的） | 高い（影響範囲が広い） |
| 根本解決 | 対症療法（再発リスクあり） | 再発防止 |
| 工数 | 小 | 大 |
| 推奨場面 | 本番緊急対応・リリース直前 | 計画的なスプリント・技術的負債解消 |

## Step by Step ガイド

### Step 1: 現象の確認 (Issue Confirmation)

**目的**: 報告内容・発生環境・期待値を正確に把握し、調査に必要な情報が揃っているか確認する

**AIのタスク**: `/spec--backlog-summary-command` の読解モードでissueKeyを指定して課題取得
```
例: PROJ-456 の課題内容を読み込んで要約してください
```

**必要な情報チェックリスト**:
- [ ] 発生環境（OS・ブラウザ・バージョン等）
- [ ] 再現手順（具体的なステップ）
- [ ] 期待値と実際の動作
- [ ] エラーメッセージ・スタックトレース
- [ ] 発生頻度（常時・特定条件下・稀に）

情報が不足している場合は、Backlogに追記依頼のコメントを行う。

---

### Step 2: 現象の再現 (Reproduction)

**目的**: バグを確実に再現する最小テストケースを作成する

**AIのタスク**: `/debug--bug-investigation-command` でエラーログ解析・再現コード作成
```
例: このスタックトレースを解析して再現手順を推測し、
    再現するテストコードを作成してください。
    [スタックトレースをペースト]
```

**再現テストの形式**:
```typescript
// 例: 再現テスト
it('バグを再現するテスト', () => {
  // バグが発生する条件を設定
  // → このテストが赤（失敗）になることを確認する
});
```

**人間のタスク**: 再現コードが実際のバグを正しく再現しているか確認

---

### Step 3: 原因の特定 (Root Cause Analysis)

**目的**: 再現コードを使ってデバッグし、問題の根本原因を特定する

**AIのタスク**: `debug--bug-investigator` エージェントで詳細調査
- スタックトレースの解析
- 関連コードのGrep/Glob検索
- データフロー追跡（入力→処理→出力の各段階で何が起きているか）
- 水平展開: 同じパターンのバグが他にも存在しないか確認

**出力**: バグ調査レポート（現象・根本原因・影響範囲・原因箇所のファイルパスと行番号）

---

### Step 4: 修正方針の策定 (Fix Strategy)

**目的**: 対症療法ではなく、根本解決を目指す方針を決める

**AIのタスク**: Quick Fix / Deep Fix を比較表形式で提示

**人間の判断項目**:
- 緊急度: 本番で障害が出ているか？ユーザー影響は何件か？
- リスク: Deep Fix をやると他のどこかが壊れる可能性は？
- スプリント: 今のスプリントで根本解決できるか？

**判断の目安**:
- 本番障害 → まず Quick Fix でリリース → 次スプリントで Deep Fix
- 開発中バグ → Deep Fix を優先

---

### Step 5: 修正と検証 (Fix & Verify)

**目的**: TDDサイクルで修正を行い、バグが解消されたことを確認する

**AIのタスク**: `/test--tdd-command` でTDDサイクル

```
🔴 Red   → Step 2で作った再現テストが失敗することを確認（バグが存在する状態）
🟢 Green → 修正コードを実装し、再現テストが通ることを確認
🔵 Refactor → 修正コードを整理し、既存テストがすべて通ることを確認
```

**水平展開**: 同じパターンのバグが他にもないか確認し、まとめて修正する
