# 設計・実装ワークフロー

クリーンアーキテクチャ・TDD・リファクタリングを組み合わせた設計・実装の進め方ガイドです。

## 人間 / AI 役割分担

| 役割 | 担当 |
|------|------|
| **アーキテクチャの意思決定**: AIが提示した設計案から、プロジェクトの文脈に最適なものを選択する | 人間 |
| **コードレビューと承認**: 生成されたロジック・テストコードが要件を満たすか確認しマージを承認する | 人間 |
| **「手戻り」の判断**: 設計ミスに気付いた時、どこまで戻って修正するかを決める | 人間 |
| **設計案の提示**: 複数のアーキテクチャ案を比較できるよう提示する | AI |
| **複雑なロジックの実装**: ドメイン知識が必要なコアロジックの実装案を提示する | AI |
| **設計パターンの提案**: GoF/DDD等のパターンに基づき最適な設計案を提示する | AI |
| **過剰設計の指摘 (YAGNI)**: 不要な複雑さや使われない機能の可能性を指摘する | AI |
| **コード品質の計測**: 循環的複雑度・メソッド長等を計測しリファクタリング優先度を示す | AI |
| **TDDサイクルの支援**: テストケース提示・Red-Green-Refactorの各フェーズを支援する | AI |
| **リファクタリング提案**: コードスメルを指摘し、修正案を提示する | AI |

---

## Step by Step ガイド

### Step 1: 作業計画の策定

**目的**: 実装対象を明確にし、変更範囲を洗い出す

**AIのタスク**: `/arch--architect-command` で設計案を複数提示
```
例: 「ユーザー認証機能を追加したい。
     技術スタック: TypeScript + NestJS + PostgreSQL」
```

**確認項目**:
- 実装対象の機能・API・UIコンポーネントの特定
- 変更が必要なファイル・モジュールの洗い出し
- 外部依存（API・DB・ライブラリ）の特定

---

### Step 2: 計画のリファクタリング（設計レビュー）

**目的**: 実装前に計画をブラッシュアップし、落とし穴を潰す

**AIとの壁打ちチェックリスト**:
- [ ] フレームワークの設計思想（CoC等）に沿っているか？
- [ ] 最適な設計パターン（GoF, DDD, Clean Architecture等）を適用できるか？
- [ ] 既存コードにリファクタリングの余地はないか？
- [ ] アーキテクチャ全体と整合しているか？
- [ ] 過剰な機能（YAGNI）はないか？
- [ ] TDD可能な設計か（依存性注入、インターフェース分離）？

---

### Step 3: TDDによる実装

**目的**: テストファーストで品質を確保しながら実装する

**TDD可能な場合**: `/test--tdd-command` でRed-Green-Refactorサイクル

```
🔴 Red   → 失敗するテストを書く
🟢 Green → テストを通過する最小限のコードを書く
🔵 Refactor → コードを改善しながらテストを維持する
```

**TDD困難な場合**（UIレンダリング・外部API連携等）:
1. プロトタイピングで動作確認
2. 実装が安定したら後追いでテスト追加

---

### Step 4: 実装後のリファクタリング

**目的**: 動くコードを「読みやすいコード」に仕上げる

**AIのタスク**: `/quality--refactor-command` で最終整理

確認項目:
- 関数・メソッドの長さ（目安: 20行以内）
- 重複コードの除去（DRY原則）
- 命名規則の一貫性
- コメントの適切さ（WHYを書く、WHATは書かない）

---

### Step 5: 人間による最終レビュー (Human Review)

**目的**: 完成物がすべての基準を満たすか確認してマージ判断を下す

**AIのタスク**: `/review--pr-review-command` でPR差分を自動レビュー

**人間の確認項目**:
- 完成したコードが要件を満たすか
- テストが適切に書かれているか
- ドキュメント（コメント・README）が更新されているか
- コードスタイル・規約に沿っているか
