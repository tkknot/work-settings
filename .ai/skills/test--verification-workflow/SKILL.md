# 検証方法作成ワークフロー

テスト設計技法を活用した、QA担当者にも伝わる検証方法の作成ガイドです。

## 人間 / AI 役割分担

| 役割 | 担当 |
|------|------|
| **「何をテストすべきか」の決定**: ビジネス価値に直結する重要シナリオを選定する | 人間 |
| **設計技法の承認**: AIが提案した技法が対象の性質に合っているか判断する | 人間 |
| **用語・表現のチェック**: コード識別子・技術用語が混入していないか最終確認する | 人間 |
| **自動化判断**: どれをE2E自動化し、どれを手動テストで残すか決める | 人間 |
| **エッジケースの追加**: 業務特有のレアケースや複雑な条件を追加する | 人間 |
| **品質基準の決定**: 「どのブラウザでOKか」「実行時間は何分以内か」を決める | 人間 |
| **設計技法の提案**: 対象の性質を分析し最適な技法を提案する | AI |
| **シナリオの網羅的列挙**: 選定技法に基づきパターンを漏れなくリストアップする | AI |
| **テストケースの自己レビューと表現改善**: 生成後に自らレビューし技術用語を業務用語に置換する | AI |
| **E2Eテストコード生成**: 自動化対象ケースをPlaywright等のスクリプトとして生成する | AI |
| **テストデータの準備**: テストに必要なデータ投入スクリプトを作成する | AI |

---

## テスト設計技法ガイド

| 技法 | 適用場面 | 特徴 |
|------|---------|------|
| **同値分析** | 入力フォーム、バリデーション | 同じ振る舞いをするグループに分類し、代表値でテストする |
| **境界値分析** | 数値範囲、文字数制限 | 最小値・最大値・境界±1に絞ってテストする |
| **ディシジョンテーブル** | 権限・状態フラグの複合条件 | 複数条件の組み合わせを表形式で整理する |
| **ペアワイズ** | 多数のオプション・パラメータ | 2因子間の組み合わせを網羅しつつテスト数を削減する |

## Step by Step ガイド

### Step 1: テストシナリオの洗い出し

**目的**: 要件定義（Backlog）に基づき、ユーザーが達成すべきゴールを列挙する

**AIのタスク**: `/test--test-case-command` でシナリオリストを提案
```
例: 「ログイン機能のテストシナリオを洗い出してください。
     対象: メールアドレス・パスワードによる認証」
```

**出力例**:
- ハッピーパス: 正しいメールアドレスとパスワードでログイン成功
- 異常系: 誤ったパスワードでログイン失敗（3回でロック）
- 異常系: 存在しないメールアドレスでログイン試行
- エッジケース: SQLインジェクション文字列を入力

**人間のタスク**: ビジネス価値に直結するシナリオを選定（全部テストはコスト過多）

---

### Step 2: テスト設計技法の選定とケース設計

**目的**: 選定した技法でテストケースをCSV形式で設計する

**フロー**:
1. AIが対象の性質を分析して技法を提案
2. 人間が技法選定を承認
3. AIが選定技法に沿ったテストケースをCSV形式で作成

**CSVフォーマット**:
```
テストID,シナリオ,テスト手順,テストデータ,期待結果,優先度,技法
TC-001,正常ログイン,1.ログイン画面を開く...,email:test@example.com,ダッシュボードに遷移,高,同値分析
```

---

### Step 3: 用語・表現のレビュー

**目的**: QA担当者や非エンジニアに伝わる表現に整える

**AIのタスク**: `/test--terminology-review-command` で技術用語を業務用語に置換

検出・置換対象:
| 技術的表現（NG） | 業務用語（OK） |
|----------------|--------------|
| `userId = 1` | ユーザーID: 1 |
| `is_active = true` | 有効状態 |
| `HTTP 200` | 正常応答 |
| `null` | 未入力 |

**人間のタスク**: AIの自己レビュー後も残っている伝わりにくい表現を最終確認し、追加修正を指示

---

### Step 4: E2E自動化の実装

**目的**: 設計したテストケースのうち自動化可能なものをPlaywrightスクリプト化する

**人間の自動化判断**:
| 自動化すべき | 手動テストで残す |
|------------|--------------|
| 繰り返し実行するリグレッション | 視覚的確認が必要なもの |
| データドリブンなパターン | 探索的テスト |
| CI/CDに組み込む一般フロー | 環境依存の操作 |

**AIのタスク**: `/test--e2e-command` でPlaywrightコード生成
```
例: 「TC-001〜TC-003をPlaywrightのE2Eテストとして実装してください」
```

---

### Step 5: レビューと実行

**目的**: 実装されたテストが正しく動作し、要件を網羅しているか確認する

**確認項目**:
- テストが正常に実行されるか
- テストの意図が正しいか（誤検知・偽陽性がないか）
- 過剰・不足がないか
- CI/CDパイプラインへの組み込みが完了しているか
