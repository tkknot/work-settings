# 要件定義ワークフロー

Backlog連携を軸とした要件定義の進め方ガイドです。上位者主導（トップダウン）・開発者主導（ボトムアップ）どちらのパターンにも対応します。

## 人間 / AI 役割分担

| 役割 | 担当 |
|------|------|
| **実行判断**: 提案された選択肢から最善策を決定し、要件として確定する | 人間 |
| **価値の定義**: それを作って誰が喜ぶか、本当に役に立つかを考える | 人間 |
| **責任の所在**: リリースして問題が起きた時に責任を取れるか判断する | 人間 |
| **コンテキスト理解**: 書類に書かれていないチームの状況・歴史を考慮する | 人間 |
| **直感的な違和感**: 「なんか変だな？」という勘や経験則を大切にする | 人間 |
| **Backlogの特定・読解**: 関連課題をMCP経由で検索し内容を整理する | AI |
| **論理的な矛盾の指摘**: 仕様書や既存コードとの矛盾点を指摘する | AI |
| **世間一般との乖離チェック**: 業界標準やUX慣習と比較して指摘する | AI |
| **抜け漏れのチェック**: エラー時・スマホ・権限差異などのケースを洗い出す | AI |
| **選択肢の整理**: 理想・現実・最小の3プランを比較できるようにまとめる | AI |
| **影響の調査**: コードベースを検索し、変更の影響範囲を特定する | AI |
| **文章の清書**: Backlogにそのまま貼れる文章として出力する | AI |

## Step by Step ガイド

### Step 1: Backlogの特定・確認

**目的**: 議論のテーマに関連する既存課題を把握し、重複や過去の議論を確認する

**AIのタスク**: `/spec--backlog-summary-command` を検索モードで実行
```
例: 「ログイン機能の改修」に関する既存課題を検索してください
```

**人間のタスク**:
- 検索結果から関連課題を選定
- 過去の議論で前提になっている決定事項を確認

---

### Step 2: 要件の具体化と壁打ち

**目的**: Why（解決課題）/ Who（ターゲット）/ Value（価値）を明確にする

**確認項目**:
- **解決課題（Why）**: このタスクは何の問題を解決するか？
- **ターゲット（Who）**: 誰がこの機能を使うか？
- **価値（Value）**: 完成したら誰がどう嬉しいか？
- **非機能要件**: パフォーマンス・セキュリティ・可用性の要件はあるか？

**AIとの壁打ち例**:
```
この仕様を実現するにあたり、以下を確認したいです：
- ユーザーが達成したいゴールは何ですか？
- 非機能要件（レスポンス時間、同時接続数等）はありますか？
- 既存の類似機能との違いは何ですか？
```

---

### Step 3: プラン策定と影響調査

**目的**: 実現可能な複数プランを比較できる状態にする

**松竹梅プランの考え方**:
| プラン | 内容 | 判断軸 |
|--------|------|--------|
| 松（理想） | 完全な解決策。長期的に最適 | 時間・コストに余裕がある場合 |
| 竹（現実） | バランスの良い解決策 | 通常のスプリントで選択 |
| 梅（最小） | とりあえず動く最小実装 | 緊急・実験・PoC |

**AIのタスク**:
- コードベース検索で影響範囲を調査
- 既存データとの整合性（マイグレーション要否）を確認
- 類似実装がないか調査

---

### Step 4: 妥当性と懸念点の総点検

**目的**: 見落としを潰し、GoサインをかけるBeforeの状態にする

**AIのタスク**: `/spec--requirements-review-command` を実行

4観点でのレビュー:
1. **論理整合性**: 矛盾・循環参照・前提条件の不整合がないか
2. **網羅性**: エラー時・スマホ・権限差異・端末差異のケースが考慮されているか
3. **業界標準との照合**: UX慣習・セキュリティ標準と乖離していないか
4. **実現可能性**: 技術制約・パフォーマンス影響・データ整合性の問題がないか

---

### Step 5: 最終確認 (Human Check)

**人間がGoサインを出す**

**Backlogへの登録**:
```
/spec--backlog-summary-command を清書モードで実行し、
確定した要件をBacklog Markdown形式で出力してもらう
```

## 関連ツール

| ツール | 用途 |
|--------|------|
| `/spec--backlog-summary-command` | Backlog課題の検索・読解・清書（3モード） |
| `/spec--requirements-review-command` | 要件の4観点レビュー |
| `spec--backlog-integration` スキル | Backlog MCPツールの使い方リファレンス |
