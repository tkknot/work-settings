# バグ調査コマンド

バグ報告を受け取り、根本原因の特定から修正案の提示まで体系的に実施します。

---

## Phase 1: 課題取得と現象把握

- BacklogのissueKeyが指定された場合: `get_issue` と `get_issue_comments` でMCP経由取得する
- それ以外: ユーザーからの報告内容・エラーログを入力として受け取る

以下の確認項目を整理する:

| 項目 | 内容 |
|------|------|
| 発生環境 | OS、ブラウザ、バージョン等 |
| 再現頻度 | 常に / 特定条件下 / 稀に |
| 期待値と実際の動作 | 何を期待し、何が起きたか |
| エラーメッセージ / スタックトレース | 全文を確認 |

---

## Phase 2: 再現調査

1. エラーログ・スタックトレースを解析して原因候補を絞り込む
2. Grep/Glob で関連コードを検索し、データフローを追跡する
3. バグを再現する最小テストケースを作成する

---

## Phase 3: 根本原因の特定

- コードリーディングで仮説を立て、検証する
- 影響範囲を特定する
  - 直接影響: バグが直接引き起こす問題箇所
  - 間接影響: 同パターンの類似バグが存在しないか水平展開で確認

---

## Phase 4: 修正方針の提案

比較表形式で出力する:

| 観点 | Quick Fix | Deep Fix |
|------|-----------|----------|
| 修正内容 | ... | ... |
| リスク | ... | ... |
| 工数 | ... | ... |
| 推奨場面 | ... | ... |

---

## Phase 5: 最終レポート

`debug--bug-investigator` の出力フォーマットに準じてレポートを出力する。
