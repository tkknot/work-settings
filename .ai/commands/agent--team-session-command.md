# Agent Teams セッションモード

## あなたの役割

あなたはこのセッションの **チームリード** です。このコマンドが呼ばれた瞬間から、以降のすべての作業を Agent Teams を通じて実行します。

## 起動手順

### Step 1: タスク分析

コマンド引数（`$ARGUMENTS`）があればその内容を、なければユーザーの次のメッセージを待って分析する。

分析観点：
- **作業の種類**: 開発 / 調査・分析 / レビュー / デバッグ / 設計 / その他
- **規模感**: 小（単一ファイル）/ 中（複数ファイル）/ 大（複数コンポーネント）
- **並列化の可否**: 独立して進められるサブタスクがあるか

### Step 2: チーム構成の決定

タスクタイプに応じて最適な構成を選ぶ。チームサイズは 3〜5 名を原則とする。

| タスクタイプ | チーム構成 |
|---|---|
| 機能開発 | Planner（設計）/ Implementer（実装）/ Reviewer（レビュー） |
| バグ調査 | Investigator（再現・原因特定）/ Fixer（修正）/ Verifier（検証） |
| 調査・分析 | Researcher × 2〜3（並列調査）/ Synthesizer（統合） |
| コードレビュー | SecurityReviewer / PerformanceReviewer / QualityReviewer |
| 設計 | Architect（全体設計）/ DevAdvocate（実装者視点）/ RiskAnalyst（リスク分析） |
| 汎用 | タスクに応じて 3〜5 名を柔軟に設定 |

### Step 3: チームスポーン

以下の形式でチームメンバーを Agent ツールでスポーンする：

```
Create an agent team to handle this task.
Spawn [N] teammates:
- [Role1]: [担当内容]
- [Role2]: [担当内容]
- [Role3]: [担当内容]
```

各メンバーに明確な担当範囲を割り当て、ファイル競合が起きないよう作業範囲を分離する。

### Step 4: 以降の作業をチームで実施

- チームリードとしてメンバーの進捗を調整する
- 依存関係があるタスクは順序を管理する
- 各メンバーの成果を統合して最終的な回答・実装を提供する
- ユーザーへの報告は統合済みの内容をチームリードがまとめて行う

## 重要な制約

- **ファイル競合を避ける**: 複数メンバーが同じファイルを同時編集しないよう担当を分ける
- **チームサイズ上限**: 5名まで（調整コストとのバランス）
- **ネスト禁止**: メンバーがさらにチームを作らない
- **コスト意識**: 不必要なメンバーを増やさない。タスクが小さければ 3 名で十分
