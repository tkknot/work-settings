# プレフィックス選定ガイド

## 概要

`.ai/skills/` に新しいスキルディレクトリを作る際、どの `{prefix}--` を使うか（既存に収めるか、新設するか）を判断するための指針。
プレフィックスの一覧・命名フォーマットの正本は `file-naming-convention.md` を参照する。

## 選定の判断軸

1. **領域・主題軸で統一する**
   既存プレフィックスはすべて「対象領域」で命名されている（`test--` `git--` `review--` `arch--` `team--` `security--` など）。新設する場合もこの軸に揃え、「何をするか（機能）」ではなく「何に関するスキルか（領域）」で名付けることを基本とする。

2. **新設より既存活用を先に検討する**
   新しいプレフィックスを増やす前に、適合する既存プレフィックスがないか確認する。例: `arch--` は「システム設計・技術選定」を含むため、"アプローチ選定" のような選定系スキルも収まりうる。

3. **既存コマンド名・予約語との衝突を避ける**
   ツールのコマンド名と紛らわしいプレフィックスは選ばない。例: `agent--` は `claude agents` / `/agents` コマンドと混同するため見送った。

4. **命名規則を満たす**（`file-naming-convention.md` ルール3）
   小文字英単語、短く直感的な名前にする。

5. **機能軸はトレードオフを理解して使う**
   `judge--`（判断）/ `select--`（選定）のような「何をするか」軸も選択肢だが、既存の領域軸との一貫性は崩れる。集約のメリットが上回るときのみ採用する。

## ケーススタディ: エージェント実行アプローチ選定スキル

Claude Code の並列実行アプローチ（サブエージェント / エージェントビュー / エージェントチーム / 動的ワークフロー）の選定を扱うスキルのプレフィックスを検討した際の経緯。

検討の流れ:

- 当初 `agent--` を候補にしたが、`claude agents` / `/agents` コマンドと紛らわしいため見送り（判断軸3）。
- 既存 `arch--`（技術選定）への収容も検討した（判断軸2）。
- 「複数候補から選ぶ」機能を端的に表す `select--`（選定）も機能軸の候補に挙げた（判断軸5）。

最終的な判断:

- **このスキル自体の新規作成を見送った**ため、新しいプレフィックスは追加していない。
- ただし「もし新設するなら」の結論としては、機能軸の `select--`（選定）が最有力という整理に至った（将来同種の選定スキルを集約できる点を評価）。
