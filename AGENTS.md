# エージェントガイドライン

英語で思考し、日本語で回答すること。

## 問題解決

- 何かうまくいかない場合は、すぐに立ち止まって再計画する。
- タスク完了前に、自分の成果をセルフレビューする。

## 作業の進め方の相談

- 計画の進め方やタスクの進め方を相談されたら、いきなり実装に入らず `plan--parallel-strategy` スキルを参照する。
- 4 つの並列実行アプローチ（サブエージェント / エージェントビュー / エージェントチーム / 動的ワークフロー）から最適なものを提案する。

## エラー処理とリカバリ

- 失敗した操作をリトライする前に、根本原因を診断する。盲目的にリトライしない。
- 不明な点がある場合は、仮定で進めるのではなく、明確化のための質問をする。
- ロジックエラーにはアプローチを変更する。一時的なエラー（ネットワーク、ファイルロック）にはリトライする。

## ヒューマン・イン・ザ・ループ

- レスポンスの最後に、人間の判断が必要な決定やトレードオフを明確にリストする。

## リポジトリ: work-settings

開発ツールの設定を一元管理するdotfilesリポジトリ。

### 構成

| ディレクトリ/ファイル | 説明 |
|---|---|
| `.ai/` | AIスキル・コマンド・ルールの定義 |
| `.claude/` | Claude Code固有の設定（settings.json） |
| `nvim/` | Neovim設定（LazyVimベース） |
| `wezterm/` | WezTerm設定 |
| `shell/` | シェル統合スクリプト（WezTerm の OSC 7/133/1337） |
| `sync_*.sh` | 各ツールの設定をホームディレクトリに配布するスクリプト |

### `.ai/` のファイル命名規則

`.ai/skills/` 配下は `{prefix}--{name}` 形式で命名する。
詳細は `.ai/rules/file-naming-convention.md` を参照。

### 共通ルール（`.ai/rules/`）

設定・スクリプトを扱う際は以下のルールに従う。

- セキュリティ・ベースライン: `.ai/rules/security-baseline.md`
  （秘密情報を含めない／状態ファイルと設定ファイルを混在させない／破壊的操作前に確認）
- コミットメッセージ規約: `.ai/rules/commit-convention.md`

### ローカル作業ファイル

`_local.` prefixのファイルはgitignoreで無視される。会社環境でのメモやスクラッチコードに使用する。
例: `_local.notes.md`, `_local.scratch.py`
